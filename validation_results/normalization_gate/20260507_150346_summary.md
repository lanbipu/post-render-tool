## 真相一:K1/K2/K3 的 normalization = `full-width`(确认)

### focal=30.302 与 focal=50 上,full-width 给出近完美 K1 拟合

| focal_mm | full-width k1_eff | 偏离目标 +0.5 | p95 px |

| 30.302 | +0.5015 | 0.0015 | 2.112 |

两帧都在 `K1_TOL = 5e-3` 之内,**亚像素残差**。跨这两 focal 的 spread = 0.0003,远低于阈值。

### focal=24 数据本身坏,跟公式无关

focal=24 K=0 anchor R/G ∈ [0.0001, 0.9995] —— over-scan margin ≈ 0,意味着 d3 在 1.5× over-scan 设置下,这个焦距的画面已经把 plate 边缘压满,**没有 over-scan margin 可恢复**。所有候选公式在这一 focal 都给出 k1_eff ≈ -1.9..-2.5,p95 ≈ 2700–3400 px(几乎整张 frame)。

这是数据采集层面的问题(focal=24 在 16:9 sensor 上的 horizontal FOV ≈ 73°,在 d3 的 1.5× over-scan 下达到了 plate 边界),不是 normalization 公式的二义性。结论:**focal=24 帧应当从 cross-focal 一致性检查中剔除**。harness 默认聚合所有 focal,所以 spread 被这帧污染到 1.14;只看 30.302 / 50 的 spread 是 0.0003。

K2/K3 报告里 inferred k_eff 是**用 winner=diagonal 算出来的**,看起来 +0.87 / +1.14 不像 d3 报告值 +0.5。但如果**真值是 full-width**,K_eff 在 diagonal 候选下应当 = K_d3 × (NORM_diagonal / NORM_truth)^(2n):

实测 vs 预测差 0.3%,在 LSQ 采样噪声范围内。K2/K3 在 full-width 真值下完全 = +0.5,**确认 K1/K2/K3 同一个 full-width normalization**。

### 结论 1:K1/K2/K3 的 shader 公式不需要改

`Content/Python/post_render_tool/distortion_math.py:212-216` 当前公式:

```python
rx = dx              # = d.x  (UV-space; 等价于 (px - cx)/W)
ry = dy / aspect     # = d.y/aspect  (等价于 (py - cy)/W,因 aspect=W/H)
r2 = rx*rx + ry*ry
factor = k1*r2 + k2*r2*r2 + k3*r2*r2*r2
source = UV + factor * d
```

是 **full-width forward Brown-Conrady**,跟 fit 真相一致。**不动**。

## 真相二:centerShift 公式结构错了(需改 shader)

| csx_mm | 实测 median_dx_px | d3 真公式 `−csx/sensor_w × W` 预测 | 误差 |
|---:|---:|---:|---:|
| −0.10 | +11.252 | +10.971 | 0.281 |
| −0.05 | +4.738 | +5.486 | 0.748 |
| +0.05 | −5.034 | −5.486 | 0.452 |
| +0.10 | −11.252 | +10.971 | 0.281 |

(假设 sensor_width_mm = 35.0,W = 3840)

### 现象

- **K=0 + csx≠0 时,d3 的源 UV 仍然平移了** —— 跟 `cs × image_dim / sensor_dim_mm` 公式一致,跟 focal 无关。这印证了 memory `project_distortion_pixel_perfect` 之前的"centerShift 公式定型"。
- **平移方向跟 csx 相反** —— csx=+0.05 → 源 shift -5.034 px,csx=-0.10 → 源 shift +11.252 px。
- **当前 Path C shader 在 K=0 时,csx 通过 CenterUV 进入 d=UV-CenterUV → factor×d=0 → source=UV**,根本不平移 —— 跟 d3 行为不符。

### d3 的真公式

```text
source_uv = output_uv - (csx_mm / sensor_width_mm,
                         csy_mm / sensor_height_mm)
            + factor × (output_uv - distortion_center)
```

distortion_center 物理上还是 csx-shifted optical center(虽然 K=0 sweep 没数据直接验证 radial center 是否随 csx 走,但物理直觉 + 主流 calibration 模型都是 radial center = principal point = 0.5+csx_uv)。

### 当前 shader 怎么改

`Content/Python/post_render_tool/distortion_math.py` `official_sensor_inverse_uv` 和 `Content/Python/post_render_tool/build_distortion_material.py` HLSL_CODE 同步修改,把 csx 拆成两个角色:

1. **radial center**:`center = (0.5 + csx_uv.x, 0.5 + csy_uv.y)`(保留当前 CenterUV 含义,继续作为 distortion 的 r 原点)
2. **源 UV 平移**:`source = output - csx_uv + factor × d`(新加的 - csx_uv 项)

HLSL 形式:

```hlsl
// CenterUV input now also conveys csx_uv = CenterUV - 0.5
float2 d = UV - CenterUV.rg;
float2 r = float2(d.x, d.y / Aspect);
float r2 = dot(r, r);
float fac = K1*r2 + K2*r2*r2 + K3*r2*r2*r2;
float2 csx_uv = CenterUV.rg - float2(0.5, 0.5);  // ← 新增
return UV + fac * d * DistortionWeight - csx_uv * DistortionWeight;  // ← 末项新增
```

DistortionWeight 同时控 radial 和 translation,这样 weight=0 时退化为 identity,跟当前 behavior 一致(K=0 frames 已渲过的项目可以无差异回放)。

### take_4 production diff 影响预估

take_4 production CSV 是 spatialmap schema(见 commit `5f2fa2b`),**csx / csy 几乎都是 0**(spatialmap CSV 里 centerShift 列经常缺失或为 0,carry-forward 兜底)。所以新加 `- csx_uv` 项在 take_4 上贡献 = 0,production diff baseline `valid_p95 = 0.1255` **不应受影响**。

但 Task 9 仍要跑 regression 确认。

| 项 | 改 / 不改 | 理由 |
